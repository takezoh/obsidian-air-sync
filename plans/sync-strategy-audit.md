# 同期戦略の監査: データロス・不要なコンフリクト・パフォーマンス問題

## 調査範囲

Smart Sync の同期エンジン全体を調査し、データロス・不要なコンフリクト・パフォーマンス劣化が発生しうるシナリオを特定。対象パイプライン: 変更検知 → エンティティ構築 → 判定エンジン → 実行 → 状態コミット。

---

## 重大な問題 (HIGH)

### 1. デルタ同期: リトライ時にローカル変更がロストする

**ファイル:** `service.ts:423`, `local-change-detector.ts:57-62`

**問題:** `consume()` は `changedPaths` を即座にクリアして返す。`executeSyncOnce()` がその後失敗（例: executor実行中のネットワークエラー）すると、リトライループが再度 `executeSyncOnce()` → `buildEntities()` → `consume()` を呼ぶ。この時 `consume()` は **空のSet** を返す（active=true だが paths はクリア済み）。

**シナリオ:**
1. ユーザーがファイル A, B, C をローカルで変更
2. 同期開始、`consume()` が `{A, B, C}` を返し、内部セットをクリア
3. Executor が A のプッシュに成功後、B でネットワークエラー
4. リトライ: `consume()` が `{}` (空) を返す
5. デルタ同期がリモート変更のみ処理 → B と C はプッシュされない
6. トークン未コミット (正常)、スナップショット未保存 (正常)、だがローカルデルタはロスト
7. B と C は次回のフルスキャンまたはユーザーが再編集するまで未同期のまま

**影響:** 変更済みファイルがリトライ時にサイレントに同期スキップされる。永続的なデータロスではない（ファイルはディスク上に存在）が、変更が長期間未同期のままになりうる。

**修正案:**
- (a) `consume()` を非破壊的にし、成功後に `confirm()` でクリア
- (b) 失敗時に `catch` ブロックで消費済みパスを `changedPaths` にマージバック

### 2. 大量削除セーフティネット: 閾値が狭すぎる

**ファイル:** `service.ts:269-288`

**問題:** 安全条件は `deletePropagateCount > 5 && deletePropagateCount === localFileCount`。つまり **全ての** ローカルファイルが削除される場合のみ発動する。Drive API の不具合で部分的なリスティングが返された場合（例: ファイルの90%のみ）、vault の大部分が警告なしに削除される。

**シナリオ:**
- Vault に100ファイル。Drive API がページネーションバグや部分的障害で10ファイルのみ返す
- 90ファイルが `remote_deleted_propagate` と判定される
- セーフティネットは発動しない（90 !== 100）
- ローカルの90ファイルが削除される

**修正案:** 割合ベースの閾値を追加（例: ローカルファイルの50%超が削除される場合に中止）。既存の絶対値チェックと併用。

## 中程度の問題 (MEDIUM)

### 3. 軽量インクリメンタル: リモート削除があるとフルスキャン強制

**ファイル:** `incremental-sync.ts:179-187`

**問題:** `applyIncrementalChangesLightweight()` は `removed` または `trashed` の変更が1件でもあると `needsFullScan: true` を返して中止する。パス解決にフル `idToPath` キャッシュが必要だが、軽量モードではロードしないため。

**影響:** Drive からのファイル削除が定期的にある vault では、デルタ同期が事実上無効化される — 毎回フルスキャンになる。インクリメンタル同期のパフォーマンスメリットが無効化される。

**修正案:** 最小限の `idToPath` マッピング（変更されたIDのみ）をロードするか、フォルダ階層メタデータに削除ファイルIDを追跡する。

### 4. リネーム = 削除 + 作成（帯域浪費と潜在的コンフリクト）

**ファイル:** エンジン設計（明示的なリネーム検出なし）

**問題:** リネームは `旧パス削除` + `新パス作成` として検出される。これにより:
- ファイルコンテンツの全再アップロード（帯域浪費）
- 旧パスに SyncRecord がある場合、エンジンは `remote_deleted_propagate`(旧) + `local_created_push`(新) と判定
- マルチデバイスシナリオ: デバイスAがリネーム、デバイスBが元ファイルを編集 → Bで旧ファイルが削除され、編集内容がロスト

**影響:** 動作は正しいが非効率。マルチデバイスでのリネーム+編集の競合で予期しない削除が発生しうる。

### 5. 三方向マージが "Ask" 戦略をサイレントにバイパス

**ファイル:** `executor-ops.ts:101-109`

**問題:** `enableThreeWayMerge` が true の場合、`ask` または `keep_newer` 戦略のコンフリクトが自動的に `three_way_merge` に昇格される。ユーザーの `ask` コールバックは三方向マージが失敗した場合のみ呼ばれる。「ask」を選択してすべてのコンフリクトをレビューしたいユーザーは、自動マージされたコンフリクトを確認できない。

**影響:** マージされたファイルに予期しない変更が含まれる可能性。コンフリクト履歴のログで軽減されるが、ユーザー期待に反する。

### 6. `resolveEmptyHashes` の逐次I/O（初回同期）

**ファイル:** `service.ts:482-503`

**問題:** 初回同期時（prevSync なし）、同一サイズのファイルはコンテンツ比較が必要。ループが `for ... await` で逐次処理し、各ファイルのローカルとリモートのコンテンツを読み取る。

**影響:** 初回同期で同一サイズのファイルが500個ある場合、数分かかりうる（500回の逐次リモート読み取り）。バウンデッド並行性でバッチ処理すべき。

### 7. コンフリクト解決後の SyncRecord のステイル化

**ファイル:** `executor-ops.ts:153-163`

**問題:** コンフリクト解決後、`buildSyncRecord()` がローカルとリモートを再 stat する。しかし解決から再 stat の間に、別の同期やユーザー変更がファイルを変更する可能性がある。保存されたレコードが実際の同期状態を反映しない場合がある。

**影響:** 低確率だが、次回同期で誤った「変更あり」検出が発生し、不要なコンフリクトにつながりうる。

## 低優先度の問題 (LOW)

### 8. リトライ間の部分進捗トラッキングなし

**ファイル:** `service.ts:143-188`

リトライ時に `executeSyncOnce()` が全エンティティを再構築し、全判定を再実行する。試行1で正常に同期されたファイルも再評価される（更新済み SyncRecord により `no_action` になるが、リスティング/比較コストは残る）。

### 9. AsyncPool(3) のハードコード並行数

**ファイル:** `executor.ts:124`

適応的スケーリングなし。一般的な使用には保守的だが、大規模 vault や高速接続ではボトルネックになりうる。正確性の問題ではない。

### 10. スナップショットが同一mtime・同一サイズのコンテンツ変更を検出しない

**ファイル:** `local-change-detector.ts:89`

mtime とサイズが変わらない場合（極めて稀なエッジケース — 例: ファイルシステムのクロック問題）、コンテンツ変更がスタートアップ diff で検出されない。次回のフルスキャンでキャッチされる。

### 11. `persistIncrementalChanges` がファイア・アンド・フォーゲット (`void`)

**ファイル:** `incremental-sync.ts:130`

`void persistIncrementalChanges(...)` — IDB 書き込みが await なしで非同期実行される。完了前にプロセスが終了すると、インクリメンタルキャッシュ更新がロストする。次回同期で旧トークンから再適用される（無害だが無駄）。

## 推奨修正（優先順位順）

1. **Issue #1:** `LocalChangeDetector` に `restore(paths)` メソッドを追加。同期失敗時に消費済みパスをマージバック。または `consume()` を非破壊的にし、成功後に `confirm()` でクリア。

2. **Issue #2:** 割合ベースのチェックを追加: `if (deletePropagateCount > 5 && deletePropagateCount > localFileCount * 0.5)`。既存の絶対値閾値を維持しつつ、部分的な全消去もキャッチ。

3. **Issue #6:** `resolveEmptyHashes` を `AsyncPool(3)` または `Promise.all` のバッチ処理で並列化。

4. **Issues #3-5:** 設計レベルの決定 — 実装前にトレードオフを議論。

## 検証方法

- **Issue #1:** ユニットテスト: 部分的プッシュ後のネットワーク障害をモック、リトライが以前消費されたパスを含むことを検証。
- **Issue #2:** ユニットテスト: 100ファイル、90件のリモート削除、セーフティネットが発動することを検証。
- **Issue #6:** 初回同期で同一サイズファイル100個以上でのベンチマーク。
